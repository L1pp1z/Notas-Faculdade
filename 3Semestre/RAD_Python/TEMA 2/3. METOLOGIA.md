A metodologia RAD tem por objetivo fazer a entrega dos sistemas em menos tempo e com menos erros.

A aplicação da metodologia RAD gera sistemas com telas e componentes padronizados devido ao uso de ferramentas que utilizam bibliotecas e templates reutilizáveis.

>[!tip] Dica
>A RAD é mais adequada para softwares de baixa complexidade

>[!warning] Cuidado!
>A RAD nem sempre é adequada para ser aplicada a um projeto.

## Vantagens e Desvantagens (Berger &; Beynon-Davies, 2009)

### Vantagens
- <span style="color:rgb(255, 192, 0)">INTEGRAÇÃO ANTECIPADA DO SISTEMA E REDUÇÃO DE RISCOS</span>
	- Entrega rápida de protótipos funcionais
	- Revisões constantes das funcionalidades
	- Iterações frequentes e testes realizados
	- Feedback do usuário
	- Permite análise mensurável do projeto
- <span style="color:rgb(255, 192, 0)">ADAPTABILIDADE E COMPARTIMENTAÇÃO DOS COMPONENTES DO SISTEMA</span>
	- Facilidade na modificação do software
	- Qualquer mudança pode afetar o sistema
	- A Flexibilidade permite criar protótipos ao longo do processo
	- Designers e Devs criam componentes funcionais, independentes e reutilizáveis
- <span style="color:rgb(255, 192, 0)">VERSÕES ITERATIVAS E MENOR TEMPO DE COLOCAÇÃO NO MERCADO</span>
	- O uso de ferramentas que suportam o desenvolvimento fortalece as equipes a entregas protótipos prontos
	- Equipes devem ser pequenas
	- O trabalho repetitivo deve ser feito por essas ferramentas, aumentando a produtividade
	- Granularização: Quebra das tarefas por conta de iterações frequentes
	- Tarefas atribuídas conforme a especialidade e experiência de cada um
- <span style="color:rgb(255, 192, 0)">FEEDBACK CONSTANTE DO USUÁRIO</span>
	- Eficiência e qualidade do projeto aumentam com a comunicação regular entre devs e users
	- Estrutura e acesso aos componentes UI/UX aumentam a importância dos feedbacks
	- Devs apresentando os protótipos para os users ficam mais confiantes que estão no caminho certo

### Desvantagens
- <span style="color:rgb(255, 255, 0)">NECESSIDADE DE EQUIPES TECNICAMENTE MUITO QUALIFICADAS</span>
	- Depende das habilidades de modelagem de dados e processos
	- Necessita que os profissionais se adaptem rapidamente
- <span style="color:rgb(255, 255, 0)">FOCO EXIGENTE NA INTERFACE</span>
	- Clientes avaliam a qualidade de uma solução baseada na interação com o protótipo
	- Avaliação a cada nova iteração
	- Devido à rapidez do desenvolvimento, os devs podem não utilizar as melhores práticas
- <span style="color:rgb(255, 255, 0)">ALTO NÍVEL DE COMPROMETIMENTOS DAS PARTES</span>
	- O Projeto evolui à medida na qual é desenvolvido
	- Comunicação entre as partes é essencial
	- Sem comunicação, a qualidade do projeto pode ser comprometida
- <span style="color:rgb(255, 255, 0)">SISTEMAS MODULARES PARA PROJETOS DE GRANDE ESCALA</span>
	- Foco nas partes essenciais do sistema
	- Segurança, desempenho e tratamento de erros, podem ser comprometidos
	- Necessidade de ferramentas padronizadas
	- Projetos de grande escala não são adequados

<span style="color:rgb(255, 192, 0)">QUANDO APLICAR:</span>
A metodologia RAD é adequada para projetos de pequena escala com equipes otimizadas de quatro a oito pessoas. Para projetos de grande escala, a RAD pode ser aplicada, desde que sejam divididos em projetos menores e mais gerenciáveis.

--- end-column ---

<span style="color:rgb(255, 192, 0)">QUANDO NÃO APLICAR:</span>
A RAD não é adequada para desenvolvimento de sistemas críticos em tempo real; sistemas de infraestrutura muito grandes e quando os requisitos funcionais precisam ser especificados detalhadamente ainda no início do projeto. Também não é adequada quando o sistema deve interagir com outros sistemas já existentes.

--- end-multi-column
##### <span style="color:rgb(255, 192, 0)">CONDIÇÃO PARA APLICAÇÃO</span>
- Disponibilidade de Profissionais experientes e comprometidos
- Comprometimento e participação efetiva do cliente
- Cliente disposto a seguir cronogramas
- Sistema divisível em módulos
- Programas e infroestrutura adequados
- Estrutura tecnológica e cultural da empresa precisam ser adequados
